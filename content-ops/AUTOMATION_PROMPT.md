# Daily ChinaFreeWeight content run

Work only in the isolated worktree for the ChinaFreeWeight repository and target only `https://www.chinafreeweight.com`.

1. Classify open pull requests before doing any work. A pending content-review PR is only a PR whose head branch exactly matches `codex/daily-content-YYYY-MM-DD` and which contains generated files under `content-ops/runs/YYYY-MM-DD/documents/`. Framework, tooling, audit, or older manually-authored content PRs must never activate research-only mode.
2. If a genuine content-review PR is open and has no explicit human approval, refresh its research and performance evidence, revalidate its complete draft pages and Preview, update the same PR if needed, then stop. Do not open a second daily batch, but do not replace complete drafts with a candidate-only report.
3. If a genuine content-review PR has explicit human approval, record the approver, time and PR URL in each approved document; rerun all hard gates; stage the approved entities; commit and push; mark the PR ready; merge it; wait for the normal production deployment; and verify production URLs, HTTP 200, sitemap inclusion and regressions. A GitHub `APPROVED` review or the repository-owner label `content-approved` is a durable approval signal. Never infer approval from passing tests, elapsed time, or lack of comments.
4. Otherwise create `codex/daily-content-YYYY-MM-DD`, initialize the run in generate mode, verify DataForSEO account access and budget, and use cached data before paid calls. Keyword cache is 30 days and SERP cache is 7 days. Never exceed USD 2/day or USD 30/month and never add funds. If access fails and no valid cache exists, update research only and clearly report that no reviewable draft could be produced.
5. Dynamically read all public locales from `i18n/locale-registry.ts`. Research demand, content gaps and buyer questions using GSC, GA4, DataForSEO keyword suggestions/overview/site gaps/related terms/SERP, plus traceable primary and reputable public sources.
6. Score candidates using purchase intent 30%, site gap 25%, GSC opportunity 20%, product fit 10%, evidence/assets 10%, and competition 5%. Select at most five blog and five solution entities. Fewer is correct when evidence or localization quality is insufficient.
7. Generate complete, native, buyer-focused localized documents in `content-ops/runs/YYYY-MM-DD/documents` before requesting review. A title, outline, brief, keyword list, candidate card, or research report is not reviewable content and must not be presented as the day's draft output. Use the existing repository data model, routes, Header, Footer and templates. Never create standalone HTML or a second blog system. Do not change product pages or existing site architecture.
8. Every blog must meet the documented depth, structure, source, FAQ, image, internal-link and CTA gates. Every solution page without complete authorized project evidence must be an explicit configuration, procurement, or application example and must not claim a completed customer project.
9. Run `npm run content:qa`, build, lint, typecheck, unit tests, Playwright, production render checks, responsive sampling, and Lighthouse thresholds. Failed or duplicate pages go to the deferred queue; never weaken or auto-rewrite around a hard gate.
10. Create a Draft PR and Vercel Preview only after at least one complete localized page has passed every hard gate with a score of at least 92. The report must include topic metrics and sources, every localized URL/image/internal link, pass/fail/deferred reasons, daily/monthly API spend, visible-term scan results, Preview URL, and mobile sampling. A research-only run may update the candidate pool but must not be called a content-review PR.
11. Wait for explicit human approval. Before approval, never merge or deploy. After approval, follow step 3 and publish directly through the normal merge/deployment flow; there is no run-count or elapsed-time exception.

Customer-visible page fields must pass the repository forbidden-language scanner. Internal research and QA records may use technical terminology, but none of it may render in titles, descriptions, headings, body, FAQ, CTA, image metadata, social metadata, or structured data.
